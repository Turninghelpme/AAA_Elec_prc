# OpenMV 学习与对接指南

## 1. 文档目的

这份文档面向当前云台控制项目，目标是帮助我们围绕赛题要求，明确 OpenMV 侧需要重点学习和实现的内容，并和现有 STM32 工程对接起来。

当前项目的主链路已经比较清晰：

`OpenMV 视觉识别 -> UART 串口发送 -> STM32 UART4 DMA 接收 -> TIM6 周期调度 -> TIM1 PWM 驱动云台舵机`

从现有代码来看，STM32 侧已经具备以下基础能力：

- `UART4 + DMA + ReceiveToIdle` 接收视觉模块数据
- `TIM6` 周期调度控制算法
- `TIM1 PWM` 输出双路舵机控制
- 云台控制算法接口已经预留

对应代码位置：

- 串口 DMA 接收入口：`Core/Src/main.c`
- UART4 初始化：`Core/Src/usart.c`
- 云台控制逻辑：`Hardware/servo.c`

因此，当前阶段 OpenMV 的重点不是泛学视觉，而是尽快成为一个稳定、实时、可控的“视觉传感器”。

## 2. 赛题导向理解

结合题目名称“运动目标控制与自动追踪系统”和当前项目实现方式，可以将赛题相关视觉任务归纳为以下几类：

1. 识别红色或绿色目标光斑
2. 获取目标中心位置或位置误差
3. 判断目标运动方向、边缘方向或角点状态
4. 将视觉结果稳定发送给主控
5. 在连续运行中保持识别稳定，不因光照和噪声频繁失效

从控制分工上建议这样理解：

- OpenMV 负责“看见”和“提取结构化结果”
- STM32 负责“接收、调度、执行控制”

如果 OpenMV 只能输出画框效果而不能输出稳定数据，那么现有 STM32 控制链路就很难真正跑起来。

## 3. 当前 STM32 工程对 OpenMV 的实际需求

从现有代码看，主控目前更适合接收两类视觉数据。

### 3.1 目标追踪类数据

适用于红点、绿点自动追踪。

建议 OpenMV 输出：

- `target_x`
- `target_y`
- `found_flag`

这类数据适合后续对接 `Servo_Track_PID()` 这一类基于目标中心的控制方法。

### 3.2 边缘/轨迹跟随类数据

适用于巡边、沿纸张边界、沿矩形边缘运动。

现有 STM32 已经在使用这类抽象量：

- `line_err_d`
- `line_angle`
- `line_corner`

这类数据适合后续对接 `Servo_LineFollow_Task()`。

因此，OpenMV 最终最好具备两套输出模式：

1. 目标追踪模式：输出目标中心坐标
2. 边缘跟随模式：输出误差、角度、角点状态

## 4. OpenMV 必学内容总览

按优先级排序，当前最需要学习的内容如下：

1. 相机初始化与图像稳定
2. 颜色阈值调试与色块识别
3. 串口通信与二进制协议发送
4. 图像结果转控制量
5. 线、边界、矩形与角点检测
6. 实时性优化与抗干扰处理

下面逐项展开。

## 5. 第一阶段：相机初始化与图像稳定

### 5.1 学习目标

让 OpenMV 在固定环境下稳定采图，使颜色识别结果不会因为自动曝光、自动白平衡漂移而频繁变化。

### 5.2 重点内容

- `sensor.reset()`
- `sensor.set_pixformat()`
- `sensor.set_framesize()`
- `sensor.skip_frames()`
- 关闭自动增益
- 关闭自动白平衡
- 必要时关闭自动曝光

### 5.3 为什么优先级最高

如果图像本身不稳定，后面的颜色阈值、光斑识别、边缘识别都会失效。很多“识别不准”的根本原因都不是算法不行，而是相机参数在自动漂移。

### 5.4 建议掌握程度

需要达到以下标准：

- 同一个红点在连续观察中颜色表现基本一致
- 同样环境下重复上电，阈值不需要每次重调
- 帧率能满足实时控制需求

### 5.5 官方资料

- `sensor` 文档：<https://docs.openmv.io/library/omv.sensor.html>
- OpenMV IDE 使用概览：<https://docs.openmv.io/openmvcam/tutorial/openmvide_overview.html>

## 6. 第二阶段：颜色阈值与目标光斑识别

### 6.1 学习目标

稳定识别红色和绿色目标点，并输出中心坐标。

### 6.2 重点内容

- LAB 阈值调试
- `img.find_blobs()`
- 面积阈值过滤
- 像素阈值过滤
- 圆形光斑与杂散反光的区分
- 多目标时选择主目标

### 6.3 为什么和赛题强相关

赛题中的运动目标、追踪点、引导点，本质上都可以先抽象成彩色目标检测。只要稳定拿到目标中心点，主控就能先做一版闭环控制。

### 6.4 建议输出结果

每一帧至少输出：

- `cx`
- `cy`
- `area`
- `found_flag`

其中 `cx`、`cy` 是最核心数据。

### 6.5 建议掌握程度

需要达到以下标准：

- 红点和绿点能分开识别
- 环境中有反光时仍能选出目标点
- 坐标抖动不至于让云台来回振荡

### 6.6 官方资料

- `image` 文档总页：<https://docs.openmv.io/library/omv.image.html>
- OpenMV 教程入口：<https://docs.openmv.io/openmvcam/tutorial/index.html>

## 7. 第三阶段：UART 串口通信与协议设计

### 7.1 学习目标

把 OpenMV 的检测结果稳定发送给 STM32。

### 7.2 重点内容

- `pyb.UART` 或 `machine.UART`
- `uart.write()`
- `struct.pack()`
- 固定帧头与帧尾
- 固定长度帧
- 状态标志位设计

### 7.3 为什么这是关键环节

STM32 这边已经在用 `HAL_UARTEx_ReceiveToIdle_DMA()` 做串口接收，说明主控端已经在等“结构化协议数据”。如果 OpenMV 只是串口打印文本，现有工程链路无法直接稳定利用。

### 7.4 协议设计建议

建议优先使用固定长度二进制帧，而不是字符串。

示例一：目标追踪模式

```text
Byte0   : 0xA5
Byte1-2 : target_x
Byte3-4 : target_y
Byte5   : found_flag
Byte6   : 0x5A
```

示例二：边缘跟随模式

```text
Byte0   : 0xA5
Byte1-2 : err_d
Byte3-4 : angle
Byte5   : corner_flag
Byte6   : 0x5A
```

### 7.5 建议掌握程度

需要达到以下标准：

- 固定频率发包，比如 20Hz 或 50Hz
- 连续发包时 STM32 能稳定接收
- 协议字段含义清晰，方便后续扩展

### 7.6 官方资料

- `pyb.UART`：<https://docs.openmv.io/library/pyb.UART.html>
- `machine.UART`：<https://docs.openmv.io/library/machine.UART.html>

## 8. 第四阶段：从图像结果到控制量

### 8.1 学习目标

让 OpenMV 不只输出“看到了什么”，而是输出“控制上可直接利用的数据”。

### 8.2 重点内容

- 图像中心与目标中心误差计算
- `err_x`、`err_y` 定义
- 方向角估计
- 丢目标状态设计
- 简单低通滤波或均值滤波

### 8.3 为什么对你们现在的工程很重要

现在 STM32 中已经有两种控制思路：

- 基于目标坐标的追踪控制
- 基于误差和角度的边缘跟随控制

如果 OpenMV 端直接给出更接近控制需求的数据，STM32 侧控制逻辑会更简单，也更稳定。

### 8.4 建议输出格式

目标追踪模式建议输出：

- `target_x`
- `target_y`
- `found_flag`

边缘跟随模式建议输出：

- `err_d`
- `angle`
- `corner_flag`

## 9. 第五阶段：线、边界、矩形和角点检测

### 9.1 学习目标

为赛题中的沿边缘运动、矩形轨迹、纸张边界识别提供视觉能力。

### 9.2 重点内容

- `get_regression()` 做主方向提取
- `find_lines()` 和 `find_line_segments()`
- `find_rects()`
- ROI 分区处理
- 角点和边界状态判断

### 9.3 为什么这一阶段重要

如果赛题要求沿 A4 纸边缘、矩形边框或其他规则轮廓运动，那么单纯的光点识别是不够的。OpenMV 需要进一步理解“当前边界方向是什么”“是否到了角点”“下一步应该如何切换边”。

### 9.4 和当前代码的关系

当前 STM32 已经定义了以下量：

- `line_err_d`
- `line_angle`
- `line_corner`

这说明控制层已经为“视觉输出误差和角度”留出了接口。OpenMV 需要把图像里的线、边、角点转换成这些数据。

### 9.5 官方资料

- `image` 文档总页：<https://docs.openmv.io/library/omv.image.html>

## 10. 第六阶段：实时性、稳定性和抗干扰

### 10.1 学习目标

让 OpenMV 在实际赛场环境下连续稳定工作。

### 10.2 重点内容

- 限制 ROI，提高帧率
- 降低分辨率换取实时性
- 控制发送频率
- 丢失目标时的恢复策略
- 简单的结果滤波
- 避免每帧无条件发送大量无用数据

### 10.3 典型问题

如果没有做这部分优化，常见现象包括：

- 坐标抖动导致云台抖动
- 帧率下降导致控制迟滞
- 短时间丢目标后系统失控
- 串口发包过快导致接收端解析不稳

## 11. 推荐学习顺序

建议按下面顺序推进，不建议一开始就直接啃复杂赛题全流程。

### 第一步：把 OpenMV 当成稳定摄像头

完成目标：

- 能稳定采图
- 能关闭自动参数
- 能观察到颜色阈值变化规律

### 第二步：只做红点和绿点识别

完成目标：

- 稳定找到红点和绿点
- 输出目标中心坐标
- 初步排除干扰点

### 第三步：通过串口给 STM32 发坐标

完成目标：

- OpenMV 定频发 `(x, y, flag)`
- STM32 稳定接收并打印或使用

### 第四步：STM32 先实现简单目标跟踪

完成目标：

- 让云台对红点或绿点进行基本追踪
- 验证主控和视觉联动闭环已经打通

### 第五步：扩展到边缘、直线、角点

完成目标：

- 识别边缘方向
- 输出 `err + angle + corner`
- 让云台按边界或方向量控制

### 第六步：优化鲁棒性

完成目标：

- 提高帧率和稳定性
- 降低误检率
- 处理短时丢目标

## 12. 建议的阶段性实验

### 实验 1：红点识别

目标：

- 在固定背景下识别红点
- 输出 `cx`、`cy`

通过标准：

- 连续运行 1 分钟，坐标稳定

### 实验 2：绿点识别

目标：

- 单独识别绿点
- 与红点阈值区分开

通过标准：

- 红绿同时存在时仍能正确区分

### 实验 3：UART 发包

目标：

- OpenMV 发固定 7 字节或 8 字节数据

通过标准：

- STM32 端连续收到稳定有效数据

### 实验 4：目标追踪闭环

目标：

- 将 `target_x`、`target_y` 接入主控 PID

通过标准：

- 云台能跟随目标移动

### 实验 5：边缘方向识别

目标：

- 提取主边缘角度
- 输出 `angle`

通过标准：

- 不同边界方向下角度变化符合预期

### 实验 6：角点识别

目标：

- 判断是否接近边缘拐点

通过标准：

- 在矩形角点附近能稳定给出状态标志

## 13. 对当前项目最实用的结论

如果目标是最快打通项目，建议优先路线如下：

1. 学会稳定识别红点和绿点
2. 学会用 UART 给 STM32 发固定格式二进制帧
3. 先输出目标坐标，跑通追踪闭环
4. 再进一步做边缘方向和角点识别

换句话说，现阶段最重要的 OpenMV 内容不是复杂视觉算法，而是：

- 图像稳定
- 颜色识别
- 串口发包
- 控制量表达

这些一旦打通，现有 STM32 工程就真正能发挥作用。

## 14. 后续开发建议

建议后面把 OpenMV 侧代码也按模块拆开：

1. 相机初始化模块
2. 红绿目标检测模块
3. 边缘/矩形检测模块
4. 串口协议发送模块
5. 模式切换模块

同时建议 STM32 侧也逐步统一协议，避免同时维护多种零散格式。
